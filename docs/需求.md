## 1. 项目概述

### 1.1 项目名称

**机器人关节智能监测网关 (Robot Joint Intelligent Monitoring Gateway)**

### 1.2 一句话描述

在 IMX6ULL-Pro 嵌入式 Linux 平台上，用 C 语言构建一个多传感器数据采集与故障诊断系统，通过 IMU 采集振动信号、CAN 总线读取设备数据、音频播报告警，模拟工业机器人预测性维护 (Predictive Maintenance) 子系统的边缘计算网关。

### 1.3 项目目标

| 层级 | 目标 |
|------|------|
| **学习目标** | 深入掌握 Linux 系统编程 (多线程、IPC、SocketCAN、IIO、ALSA、epoll) |
| **产出目标** | 2000+ 行 C 代码、模块化传感器框架、PC 端波形监控面板 |
| **简历目标** | "基于 IMX6ULL 设计并实现了工业机器人关节健康监测边缘网关" |

### 1.4 场景叙事

```
工业现场:
  一台六轴机械臂在产线上 24h 工作
  每个关节旁边装了一个 IMX6ULL 监测网关
  网关通过:
    ├── IMU (ICM-20608)  采集关节振动/姿态 → 检测轴承磨损
    ├── CAN 总线          读取伺服驱动器温度/电流 → 检测过载
    ├── LED 三色灯        绿=正常 / 黄=预警 / 红=故障
    ├── 语音播报          "3号关节温度异常, 请检查"
    └── 以太网            上报到 MES 系统 → 历史趋势分析
  
  本项目模拟的:
    1 个监测网关 + 1 个虚拟伺服节点 (CAN loopback 自收发)
    + 1 个 IMU + 按键 + LED + 音频播放
```

## 2. 硬件资源

### 2.1 IMX6ULL-Pro 板载外设 (已实测可用)

| 外设 | 芯片/接口 | 设备节点 | 用途 |
|------|----------|---------|------|
| 6 轴 IMU | ICM-20608-G (SPI) | `/dev/iio:device1` | 振动数据采集 (加速度+角速度) |
| CAN 总线 | FlexCAN + TJA1042 | `can0` | 读取伺服节点数据 (loopback 模拟) |
| 音频输出 | WM8960 (I2S) | `hw:0,0` | 语音告警播报 |
| 用户 LED | GPIO133 | `/sys/class/gpio/gpio133` | 设备状态指示灯 |
| 用户按键 | KEY1, KEY2 (gpio-keys) | `/dev/input/event1` | 模式切换 / 告警确认 |
| 光线传感器 | AP3216C (I2C-0, 0x1e) | `/dev/i2c-0` 裸读写 | 环境光监测 (可选) |
| 双网口 | LAN8720A ×2 | `eth0`, `eth1` | 以太网通信 |
| WiFi | RTL8723BU (USB) | `wlan0` | 无线通信 (可选) |

## 3. 功能需求

### 3.1 IMU 振动采集模块

| ID | 需求描述 | 优先级 |
|----|---------|--------|
| IMU-01 | 通过 IIO 子系统读取 ICM-20608 的 3 轴加速度 (X/Y/Z) 和 3 轴角速度 | P0 |
| IMU-02 | 支持配置采样频率 (当前默认 50Hz，目标 100-200Hz) | P0 |
| IMU-03 | 计算加速度 RMS (均方根) 值，作为振动强度指标 | P1 |
| IMU-04 | 实现简易 FFT (Goertzel 算法) 检测特定频率的振动峰值 | P2 |

### 3.2 CAN 数据采集模块

| ID | 需求描述 | 优先级 |
|------|---------|--------|
| CAN-01 | 通过 SocketCAN 接口收发 CAN 帧 | P0 |
| CAN-02 | Loopback 模式下模拟一个伺服节点，周期性发送心跳帧 + 状态数据帧 (温度/电流/状态码) | P0 |
| CAN-03 | 主站接收并解析 CAN 帧，提取温度、电流、状态码字段 | P0 |
| CAN-04 | CAN 帧接收超时检测 (节点离线判断) | P1 |
| CAN-05 | 实时打印 CAN 帧统计 (帧率/错误计数) | P2 |

### 3.3 故障诊断引擎

| ID | 需求描述 | 优先级 |
|------|---------|--------|
| DIAG-01 | 三态状态机: NORMAL → WARNING → FAULT，支持告警确认 ACKED | P0 |
| DIAG-02 | 振动阈值检测: 加速度 RMS 超过阈值 → 触发 WARNING | P1 |
| DIAG-03 | CAN 数据阈值检测: 温度/电流超过阈值 → 触发 WARNING | P1 |
| DIAG-04 | 通信超时检测: CAN 节点超过 3s 无心跳 → FAULT | P1 |
| DIAG-05 | 趋势记录: 记录最近 N 次采集数据，分析变化趋势 (递增/递减/稳定) | P2 |
| DIAG-06 | 故障恢复: 连续 M 次正常读数后自动从 FAULT 恢复到 NORMAL | P2 |

### 3.4 LED 状态指示

| ID | 需求描述 | 优先级 |
|------|---------|--------|
| LED-01 | 绿色常亮 = NORMAL (系统正常) | P0 |
| LED-02 | 黄色闪烁 = WARNING (有预警) | P0 |
| LED-03 | 红色快闪 = FAULT (故障) | P0 |
| LED-04 | LED 状态与诊断引擎状态机联动 | P0 |

### 3.5 音频告警

| ID | 需求描述 | 优先级 |
|------|---------|--------|
| AUD-01 | 通过 ALSA 播放预录的 WAV 语音片段 | P1 |
| AUD-02 | NORMAL→WARNING 时播放 "注意，检测到异常振动" | P1 |
| AUD-03 | WARNING→FAULT 时播放 "警告，设备故障，请立即检查" | P1 |
| AUD-04 | 播放不阻塞主循环 (异步播放) | P1 |

### 3.6 按键交互

| ID | 需求描述 | 优先级 |
|------|---------|--------|
| BTN-01 | KEY1 短按: 切换 LED 显示模式 (状态指示 / 数据指示) | P1 |
| BTN-02 | KEY2 短按: 确认告警 (FAULT → ACKED → 恢复后回 NORMAL) | P1 |
| BTN-03 | KEY1 长按 (>2s): 触发系统自检 | P2 |

### 3.7 TCP 监控服务器

| ID | 需求描述 | 优先级 |
|------|---------|--------|
| NET-01 | TCP Server 监听 :5555，支持多客户端连接 (epoll 单线程) | P0 |
| NET-02 | 通信协议: JSON 行，每条含时间戳、来源模块、数据内容 | P0 |
| NET-03 | 客户端命令: `STATUS` (查询状态), `SENSORS` (最新传感器数据), `LOG` (最近日志) | P1 |
| NET-04 | PC 端 Python 客户端: 连接 → 接收实时数据 → matplotlib 绘制波形 | P1 |

### 3.8 数据记录

| ID | 需求描述 | 优先级 |
|------|---------|--------|
| LOG-01 | CSV 格式记录: `[时间戳, 传感器ID, X, Y, Z, 温度, 状态]` | P1 |
| LOG-02 | 日志按小时滚动 (最多保留 24 个文件) | P1 |
| LOG-03 | 故障事件单独记录到 `fault.log` | P2 |

## 4. 非功能需求

| ID | 需求描述 |
|----|---------|
| NF-01 | **代码风格**: C (GNU99)，4 空格缩进，Doxygen 注释 |
| NF-02 | **编译**: 零警告 (`-Wall -Wextra -Werror`)，支持 x86_64 (PC) 和 armv7 (IMX6ULL) |
| NF-03 | **模块化**: 每个外设一个独立 C 模块 (.h + .c)，提供统一硬件抽象接口 |
| NF-04 | **内存**: 静态分配为主，避免动态内存碎片 |
| NF-05 | **可靠性**: 单个传感器故障不影响其他模块 (降级运行) |
| NF-06 | **可移植**: 平台相关代码与业务逻辑分离 |

### 5.2 线程模型

```
main thread (主线程)
    │
    ├── IMU Thread (imu_thread)
    │   每 10ms: 读 /dev/iio:device1 → 写入环形缓冲区 → 计算 RMS
    │
    ├── CAN Thread (can_thread)
    │   循环: can0 recv → 解析帧 → 写入消息队列
    │   同时: 每 1s 发送模拟伺服数据帧
    │
    ├── Audio Thread (audio_thread)
    │   等待播放请求 → ALSA 播放 WAV → 不阻塞主循环
    │
    ├── Button Thread (button_thread)
    │   epoll 监听 /dev/input/event1 → 短按/长按识别 → 事件入队
    │
    ├── TCP Server Thread (net_thread)
    │   epoll 监听 :5555 → 接受连接 → 收发 JSON
    │
    └── Main Loop (main thread)
        epoll 消费消息队列 → 诊断引擎判断 → 更新 LED + 触发语音
```

## 7. 验收标准

### 7.1 必达标准 (项目合格线)

- [ ] 5 个外设模块全部独立调通 (IMU/CAN/LED/Button/Audio)
- [ ] 多线程采集框架运行稳定，无死锁、无内存泄漏
- [ ] 诊断引擎能根据 IMU 振动数据 + CAN 数据做出 NORMAL/WARNING/FAULT 判断
- [ ] LED 颜色随状态变化，按键能确认告警
- [ ] TCP 服务器能向 PC 客户端推送实时传感器数据
- [ ] 编译零警告 (`-Wall -Wextra -Werror`)

### 7.2 优秀标准 (简历项目级)

- [ ] IMU 采样率达到 100Hz+，环形缓冲区不丢数据
- [ ] CAN 模拟节点发送的心跳帧和主站的接收解析完全正确
- [ ] 音频告警异步播放，不阻塞主循环
- [ ] PC Python 客户端实时显示 IMU 振动波形 + CAN 温度曲线
- [ ] 架构分析文档包含技术选型理由和能力边界讨论
- [ ] Doxygen 生成完整 API 文档
